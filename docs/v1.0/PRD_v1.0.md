# LocusAtlas - 空间定位生态平台 PRD

## 1. 产品愿景

打造一个多设备空间定位生态平台，让人和机器人在同一物理空间中实现"透视级"空间感知 — 指挥中心上帝视角监控所有设备位置，现场设备穿透障碍物感知彼此方位，实现人机空间协同与任务调度。

一句话：让物理空间中每一个设备和人，都有精确的坐标和彼此的感知。

## 2. 核心场景

### 场景 A：指挥中心监控
操作员在 Web 端 3D 地图上，实时看到所有在线设备（HoloLens2/Kinect等）的精确位置和状态。穿越障碍物也能看到设备在墙后、楼层间的大致位置。

### 场景 B：现场人机协作
佩戴 HoloLens2 的人员，能在 MR 视野中看到机器人的方位标记，即使被障碍物遮挡也能感知大致方向和距离。机器人也能获取人员相对位置用于避障/跟随。

### 场景 C：空间任务调度
在 Web 端标注巡检路径/目标点，下发给机器人执行。机器人按空间坐标导航，指挥中心实时追踪进度。

## 3. 系统架构

```
┌─────────────────────────────────────────────────────┐
│                    Web 平台层                         │
│  3D可视化 │ 设备管理 │ 地图管理 │ 任务标注 │ API网关  │
└──────────────────────┬──────────────────────────────┘
                       │ WebSocket / REST
┌──────────────────────┴──────────────────────────────┐
│                   定位服务层                          │
│  点云匹配引擎 │ 位姿优化 │ 多设备融合 │ 空间锚点管理  │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────┴──────────────────────────────┐
│                   设备接入层                          │
│  HoloLens2 SDK  │  Kinect SDK  │  [未来设备SDK...]   │
└─────────────────────────────────────────────────────┘
```

### 3.1 端云协同计算

| 环节 | 设备端 | 云端/服务器 |
|------|--------|------------|
| 环境感知 | HoloLens2: 自带SLAM持续跟踪；Kinect: 周期性采集点云帧 | 接收感知数据 |
| 粗定位 | IMU积分推算初始位姿 | WiFi/BLE指纹辅助（可选） |
| 精定位 | — | 点云匹配（ICP/NDT）对齐到全局地图 |
| 位姿下发 | 接收校准后的全局位姿 | 下发全局坐标 |
| 持续跟踪 | HoloLens2: 自身SLAM维持；Kinect: 周期性重新匹配 | 监控漂移，必要时触发重新对齐 |

### 3.2 定位流程

```
设备启动 → 采集环境数据 → 上传点云片段到服务器
        → 服务器与全局地图匹配（ICP/NDT）→ 得到全局位姿
        → 下发位姿给设备 → 设备进入跟踪模式
        → HoloLens2: SLAM自主跟踪，定期校准
        → Kinect: 周期性重新匹配（5-30秒，可配置）
```

## 4. 功能模块

### 4.1 地图管理
- 上传预扫描点云地图（PCD/PLY/LAS格式）
- 地图预处理流水线：去噪 → 下采样 → 体素化（原始Kinect点云过大，必须预处理后才能用于匹配和渲染）
- 地图列表管理、版本控制
- 地图预览（Web端3D渲染）
- 地图区域划分（楼层/功能区标注）

### 4.2 设备管理
- 设备注册与接入（设备类型、序列号、能力声明）
- 设备在线状态监控
- 设备分组管理（按区域/任务分组）
- 设备类型适配层（抽象接口，适配不同传感器）

### 4.3 实时定位
- 点云片段上传与匹配
- 全局位姿计算与下发
- HoloLens2: 近实时定位（SLAM + 周期校准，目标 <500ms 延迟）
- Kinect: 周期性定位（5-30秒间隔，可配置）
- 位姿漂移检测与自动修正

### 4.4 3D 可视化监控
- 三维点云地图渲染（Three.js / Potree）
- 设备位置实时标注（图标 + 方向 + 状态）
- 设备轨迹回放
- 障碍物透视模式（设备在遮挡物后的方向指示）
- 多视角切换（自由视角 / 跟随设备 / 俯视图）

### 4.5 设备间感知
- 任意两设备间相对位置计算
- 位置数据广播（设备可订阅其他设备位置）
- HoloLens2 端: MR 视野中渲染其他设备方位标记
- 机器人端: 相对位置 API 供导航/避障使用

### 4.6 API 服务
- 设备位姿查询（REST）
- 位姿实时推送（WebSocket）
- 地图数据访问
- 设备管理 CRUD

## 5. 技术选型（建议）

| 层级 | 技术 | 理由 |
|------|------|------|
| 前端3D | Three.js + Potree | Potree专做大点云渲染，Three.js做交互 |
| 前端框架 | React + TypeScript | 生态成熟，类型安全 |
| 后端 | Python (FastAPI) | 点云处理库丰富（Open3D, PCL bindings） |
| 点云匹配 | Open3D (ICP/NDT) | 成熟稳定，Python生态 |
| 实时通信 | WebSocket (Socket.IO) | 双向低延迟 |
| 数据库 | PostgreSQL + PostGIS | 空间数据友好 |
| 消息队列 | Redis Streams | 设备数据流缓冲 |
| 设备SDK | C# (HoloLens2) / C++ (Kinect) | 各平台原生语言 |

## 6. MVP 范围（1-3 设备）

### MVP 必做
- [ ] Web 端点云地图上传与 3D 渲染
- [ ] HoloLens2 接入：采集点云帧 → 上传 → 匹配 → 位姿下发 → SLAM跟踪
- [ ] Kinect 接入：周期性点云采集 → 上传 → 匹配 → 位姿返回
- [ ] Web 端实时设备位置显示（2D俯视 + 3D视角）
- [ ] 设备间相对位置计算与 API
- [ ] 基础设备管理（注册、在线状态）

### MVP 暂不做
- 空间标注/任务调度
- WiFi/BLE 指纹辅助定位
- 轨迹回放
- 多地图/多楼层切换
- 设备分组与权限

## 7. 扩展路线

| 阶段 | 能力 | 设备规模 |
|------|------|---------|
| MVP | 单场景，2种设备类型，基础定位+展示 | 1-3 |
| V1.0 | 多场景，设备间感知，轨迹记录 | 5-10 |
| V2.0 | 空间标注/任务调度，WiFi/BLE辅助 | 10-50 |
| V3.0 | 多区域联动，第三方设备SDK接入，生态开放 | 50+ |

## 8. 关键技术风险

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| 点云匹配实时性不足 | HoloLens2延迟过高 | 端云协同，设备端SLAM保跟踪，服务器只做校准 |
| ICP在特征稀疏区匹配失败 | 定位丢失 | 引入几何特征描述子（FPFH），必要时人工指定初始位姿 |
| Kinect点云传输带宽大 | 网络拥塞 | 点云下采样+体素滤波后再上传 |
| 多设备位姿坐标系不统一 | 设备间相对位置不准 | 全局统一地图坐标系，所有设备对齐到同一地图 |
| 大点云地图Web渲染卡顿 | 监控不可用 | Potree LOD渲染+八叉树索引 |

## 9. 已决议

| 问题 | 结论 | 备注 |
|------|------|------|
| 地图扫描工具 | Kinect | 同一设备既扫描地图又做定位，需加地图预处理流水线（去噪→下采样→体素化） |
| 服务器部署 | 单体服务，局域网 | 一台PC跑所有服务，1-3设备够用，后续再拆分 |
| HoloLens2 MR渲染 | MVP不做，V1.0加 | 先在Web端验证定位精度，精度够再做头盔内MR叠加，避免标记飘移 |
| 数据安全 | 静态Token认证 | 设备注册时分配固定token，局域网内足够，后续升级JWT |
| 离线模式 | MVP不做 | 断网后重连重新匹配，HoloLens2自身SLAM可短暂维持 |
